# Historia de Usuario: Registro de Etapa por Negociación

La información detallada en la historia se puede hallar en el anexo correspondiente. Para el desarrollo de esta funcionalidad, se creó el componente `ActualizarEtapaForm.jsx` utilizando React y Tailwind CSS para permitir a los agentes actualizar la etapa de una negociación a través de su ciclo de vida. Las etapas válidas son: interés, negociación, cierre, finalizada, y cancelada. El sistema registra automáticamente la fecha de cada cambio de etapa.

La interfaz fue realizada utilizando componentes de React con hooks `useState` y `useEffect`, y la librería Axios para peticiones HTTP. El componente incluye un selector de etapas que muestra todas las etapas válidas con sus etiquetas formateadas, deshabilita la etapa actual para prevenir seleccionarla de nuevo, y muestra descripciones detalladas de cada etapa cuando se selecciona una nueva. El componente valida que se seleccione una etapa diferente a la actual antes de permitir el envío. Al enviar, se envía una petición PUT a `/api/negociaciones/:id` con la nueva `etapa` en el cuerpo y el token JWT. Si la actualización es exitosa, muestra un toast de éxito mediante Sonner y ejecuta el callback `onSuccess` con la negociación actualizada.

En el backend se utilizó Node.js con Express.js y Prisma como ORM. El modelo `Negociacion` incluye el campo `etapa` de tipo `EtapaNegociacion` con default 'interes', y el campo `fecha_cambio_etapa` de tipo DateTime con default now() que se actualiza cada vez que cambia la etapa. El enum `EtapaNegociacion` define las etapas válidas: interes, negociacion, cierre, finalizada, cancelada. La función `actualizarNegociacion` en `negociacion.controller.js` valida primero los datos mediante `validationResult` de express-validator, verifica que la etapa sea válida comparándola con un array de etapas permitidas (retorna 400 si no es válida), busca la negociación en la base de datos, verifica que exista (retorna 404 si no existe), verifica que la negociación esté activa (retorna 400 si está inactiva), verifica permisos (agente solo puede actualizar sus propias negociaciones mediante `negociacion.agenteId === usuario.id`, admin no puede modificar etapas, solo visualizar, retorna 403 si es admin), y actualiza la negociación estableciendo la nueva `etapa` y `fecha_cambio_etapa` a la fecha actual usando `prisma.negociacion.update`. La función retorna la negociación actualizada con información completa del cliente, propiedad y agente mediante `include`.

La ruta PUT `/api/negociaciones/:id` fue configurada en `negociacion.routes.js` con middlewares `verificarToken` (autenticación), `esAgenteOAdmin` (autorización: solo agentes y administradores pueden acceder, aunque admin no puede modificar), y `validarActualizarNegociacion` (validación usando express-validator para validar que `etapa` sea una de las opciones válidas del enum), registrada en `index.js` bajo el prefijo `/api/negociaciones`. El sistema valida que solo el agente responsable pueda actualizar las etapas de sus negociaciones.
