# Historia de Usuario: Registro de Negociación

La información detallada en la historia se puede hallar en el anexo correspondiente. Para el desarrollo de esta funcionalidad, se creó el componente `CrearNegociacion.jsx` utilizando React y Tailwind CSS para permitir a administradores y agentes crear negociaciones vinculando un cliente con una propiedad. Las negociaciones se crean con etapa inicial "interés" y el agente asignado es automáticamente el agente responsable del cliente.

La interfaz fue realizada utilizando componentes de React con hooks `useState` y `useEffect`, y la librería Axios para peticiones HTTP. El componente modal se abre desde `PanelNegociaciones.jsx` y carga los clientes activos del agente (o todos para admin) mediante GET `/api/clientes?estado=activo`, y carga las propiedades disponibles mediante GET `/api/propiedades/negociaciones/disponibles?includeAgente=true`. El componente incluye un selector de cliente que muestra nombre, email y tipo de cliente, un selector de propiedad con búsqueda por título/ciudad/tipo, un toggle para agentes que permite filtrar solo sus propiedades propias, y visualización detallada de la propiedad seleccionada mostrando imagen, título, precio, ubicación, tipo, y si es propiedad propia o de otro agente. El formulario valida que ambos campos estén seleccionados antes de permitir el envío. Al enviar, se envía una petición POST a `/api/negociaciones` con `clienteId` y `propiedadId`, y el token JWT. Si la creación es exitosa, muestra un toast de éxito mediante Sonner, limpia el formulario, ejecuta el callback `onSuccess` para recargar la lista, y cierra el modal.

En el backend se utilizó Node.js con Express.js y Prisma como ORM. El modelo `Negociacion` fue definido en `schema.prisma` con campos: `id` (Int autoincrement), `clienteId` (Int con relación a Cliente), `propiedadId` (Int con relación a Propiedad), `agenteId` (Int con relación a Usuario agente), `etapa` (enum EtapaNegociacion con default 'interes'), `fecha_inicio` (DateTime con default now()), `fecha_cambio_etapa` (DateTime con default now()), y `activo` (Boolean con default true para soft delete). El modelo incluye un constraint único `@@unique([clienteId, propiedadId])` que previene múltiples negociaciones activas entre el mismo cliente y propiedad. El enum `EtapaNegociacion` define las etapas: interes, negociacion, cierre, finalizada, cancelada.

La función `crearNegociacion` en `negociacion.controller.js` valida primero los datos mediante `validationResult` de express-validator, verifica que el cliente exista y esté activo (retorna 400 si está inactivo), verifica que el cliente tenga un agente asignado (retorna 400 si no tiene agente), verifica permisos (agente solo puede crear negociaciones para sus propios clientes mediante `cliente.agenteId === usuario.id`, admin puede crear para cualquier cliente), verifica que la propiedad exista, verifica que la propiedad esté disponible (retorna 400 si `estado_publicacion !== 'disponible'`), verifica que no exista una negociación activa duplicada entre el mismo cliente y propiedad (retorna 400 si existe), y crea la negociación asignando automáticamente el `agenteId` del cliente (no del usuario que crea, sino del agente responsable del cliente). La función retorna la negociación creada con información completa del cliente, propiedad y agente mediante `include`.

La ruta POST `/api/negociaciones` fue configurada en `negociacion.routes.js` con middlewares `verificarToken` (autenticación), `esAgenteOAdmin` (autorización: solo agentes y administradores pueden crear negociaciones), y `validarCrearNegociacion` (validación usando express-validator para validar que `clienteId` y `propiedadId` sean enteros válidos mayores a 0), registrada en `index.js` bajo el prefijo `/api/negociaciones`.
