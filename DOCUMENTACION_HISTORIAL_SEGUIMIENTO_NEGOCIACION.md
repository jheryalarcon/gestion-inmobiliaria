# Historia de Usuario: Historial de Seguimiento por Negociación

La información detallada en la historia se puede hallar en el anexo correspondiente. Para el desarrollo de esta funcionalidad, se creó el componente `HistorialSeguimientos.jsx` utilizando React y Tailwind CSS para permitir a los agentes registrar y visualizar un historial cronológico de todas las interacciones y acciones realizadas con un cliente durante una negociación. Los seguimientos incluyen tipo de interacción (llamada, visita, mensaje, email, reunión, documento, otro), comentario descriptivo, fecha, y agente responsable.

La interfaz fue realizada utilizando componentes de React con hooks `useState` y `useEffect`, y la librería Axios para peticiones HTTP. El componente carga los seguimientos al montarse mediante GET `/api/seguimientos/:negociacionId` con el token JWT. El componente incluye un formulario para crear nuevos seguimientos que solo se muestra si el usuario es el agente responsable (verificado mediante `canCreateSeguimiento()` que compara `negociacion.agenteId === usuario.id`). El formulario incluye un selector de tipo de seguimiento con iconos y etiquetas formateadas (📞 Llamada, 🏠 Visita, 💬 Mensaje, 📧 Email, 🤝 Reunión, 📄 Documento, 📝 Otro), un textarea para el comentario con límite de 1000 caracteres y contador de caracteres, y botones de cancelar y guardar. Al enviar el formulario, se valida que el comentario no esté vacío, se envía una petición POST a `/api/seguimientos/:negociacionId` con `comentario` y `tipo`, y el token JWT. Si la creación es exitosa, muestra un toast de éxito mediante Sonner, limpia el formulario, oculta el formulario, recarga los seguimientos, y ejecuta el callback `onSeguimientoCreado` si está definido. El componente muestra la lista de seguimientos ordenados cronológicamente (más recientes primero), cada uno con icono del tipo, nombre del agente, fecha formateada, tipo como badge, y comentario. Si no hay seguimientos, muestra un mensaje motivacional. El componente incluye un mensaje informativo para administradores indicando que solo pueden visualizar.

En el backend se utilizó Node.js con Express.js y Prisma como ORM. El modelo `Seguimiento` fue definido en `schema.prisma` con campos: `id` (Int autoincrement), `negociacionId` (Int con relación a Negociacion), `agenteId` (Int con relación a Usuario agente), `fecha` (DateTime con default now()), `comentario` (String), `tipo` (enum TipoSeguimiento con default 'otro'), y `activo` (Boolean con default true para soft delete). El enum `TipoSeguimiento` define los tipos: llamada, visita, mensaje, email, reunion, documento, otro. La función `obtenerSeguimientos` en `seguimiento.controller.js` verifica que la negociación exista y esté activa, obtiene todos los seguimientos activos de esa negociación ordenados por fecha descendente, incluye información del agente mediante `include`, y retorna los seguimientos con el total. La función `crearSeguimiento` valida que el comentario esté presente y no esté vacío (retorna 400 si falta), valida que el comentario no exceda 1000 caracteres (retorna 400 si excede), verifica que la negociación exista y esté activa, verifica permisos (solo el agente responsable puede crear seguimientos mediante `negociacion.agenteId === agenteId`, retorna 403 si no tiene permisos), valida que el tipo sea válido comparándolo con un array de tipos permitidos (retorna 400 si no es válido), y crea el seguimiento con `fecha` establecida a la fecha actual usando `prisma.seguimiento.create`.

Las rutas GET `/api/seguimientos/:negociacionId` (obtener) y POST `/api/seguimientos/:negociacionId` (crear) fueron configuradas en `seguimiento.routes.js` con middlewares `verificarToken` (autenticación), `esAgenteOAdmin` (autorización: solo agentes y administradores), y `validarCrearSeguimiento` (validación usando express-validator para validar que `comentario` tenga entre 1 y 1000 caracteres y que `tipo` sea uno de los valores válidos del enum), registradas en `index.js` bajo el prefijo `/api/seguimientos`. El sistema permite solo al agente responsable registrar nuevos seguimientos, mientras que los administradores solo pueden visualizarlos.
