# Historia de Usuario: Desactivación o Reactivación de Agentes

La información detallada en la historia se puede hallar en el anexo correspondiente. Para el desarrollo de esta funcionalidad, se implementó la capacidad de cambiar el estado de los agentes inmobiliarios (activo/inactivo) mediante soft delete. Cuando un agente es desactivado, no puede acceder al sistema hasta que sea reactivado. El sistema previene que un administrador se desactive a sí mismo. Solo los administradores pueden desactivar y reactivar agentes.

Se implementaron botones de acción condicionales en `PanelAgentes.jsx` utilizando React: botón "🚫 Desactivar" para agentes activos y botón "✅ Activar" para agentes inactivos. El componente incluye estados para controlar los modales: `showDesactivarModal`, `showReactivarModal`, y `agenteSeleccionado`. Al hacer clic, se ejecutan funciones que guardan el agente seleccionado y abren modales de confirmación. Los modales muestran el nombre del agente, advertencias sobre las consecuencias de la acción, y botones de cancelar y confirmar. Al confirmar, se envía una petición PATCH a `/api/agentes/:id/estado` con `activo: false` o `activo: true` y el token JWT. Después de desactivar o reactivar exitosamente, se muestra un toast de éxito mediante Sonner, se cierra el modal, se limpia el agente seleccionado, y se recarga la lista de agentes.

En el backend se utilizó Node.js con Express.js y Prisma como ORM. El modelo `Usuario` incluye el campo `activo` (Boolean con default true) que permite implementar soft delete sin eliminar físicamente los registros. La función `actualizarEstadoAgente` en `agentes.controller.js` recibe el `id` desde los parámetros y `activo` (Boolean) desde el cuerpo de la petición. La función verifica primero que solo administradores puedan cambiar estado (retorna 403 si no es admin), busca el agente por ID y rol 'agente', verifica que el agente exista (retorna 404 si no existe), previene auto-desactivación verificando que el ID del agente no sea igual al ID del usuario autenticado (retorna 400 si intenta desactivarse a sí mismo), y actualiza el campo `activo` del agente usando `prisma.usuario.update`. La función retorna información del agente actualizado con el nuevo estado.

La ruta PATCH `/api/agentes/:id/estado` fue configurada en `agentes.routes.js` con middlewares aplicados a todas las rutas mediante `router.use`: `verificarToken` (autenticación) y `esAdmin` (autorización: solo administradores pueden cambiar el estado de agentes), registrada en `index.js` bajo el prefijo `/api/agentes`. El sistema previene auto-desactivación para mantener al menos un administrador activo en el sistema.
