# Historia de Usuario: Desactivación de Clientes

La información detallada en la historia se puede hallar en el anexo correspondiente. Para el desarrollo de esta funcionalidad, se implementó la desactivación de clientes mediante soft delete, manteniendo los datos en la base de datos pero ocultándolos de las operaciones normales. Los agentes solo pueden desactivar sus propios clientes, mientras que los administradores pueden desactivar cualquier cliente. Solo los administradores pueden reactivar clientes inactivos. El sistema registra información de auditoría sobre quién y cuándo desactivó cada cliente.

Se implementaron botones de desactivar y reactivar en `PanelClientes.jsx` utilizando React. El botón "Desactivar" aparece solo si el cliente está activo, y el botón "Reactivar" aparece solo si el cliente está inactivo. Al hacer clic, se ejecutan funciones que guardan el cliente seleccionado y abren modales de confirmación. Los modales muestran el nombre del cliente, advertencias sobre las consecuencias, y botones de cancelar y confirmar. Si el usuario confirma, se envía una petición PATCH a `/api/clientes/:id/desactivar` o `/api/clientes/:id/reactivar` con el token JWT. Si la acción es exitosa, muestra un toast de éxito mediante Sonner, cierra el modal, y recarga la lista de clientes. Si hay errores, muestra toasts con mensajes específicos según el código de error.

En el backend se utilizó Node.js con Express.js y Prisma como ORM. Se agregaron campos de auditoría al modelo `Cliente` en `schema.prisma`: `fecha_desactivacion` (DateTime opcional), `desactivado_por` (Int opcional), y relación `usuario_desactivador` con el modelo `Usuario` mediante `@relation("ClienteDesactivador")`. El campo `activo` (Boolean, default true) permite implementar soft delete sin eliminar físicamente los registros.

La función `desactivarCliente` en `cliente.controller.js` verifica que el cliente exista (retorna 404 si no existe), verifica permisos (agente solo puede desactivar sus propios clientes mediante `cliente.agenteId === usuario.id`, admin puede desactivar cualquier cliente), verifica que el cliente esté activo (retorna 400 si ya está inactivo), y actualiza el cliente estableciendo `activo=false`, `fecha_desactivacion` a la fecha actual, y `desactivado_por` al ID del usuario autenticado. La función `reactivarCliente` verifica que solo administradores puedan reactivar (retorna 403 si no es admin), verifica que el cliente esté inactivo (retorna 400 si ya está activo), y actualiza el cliente estableciendo `activo=true` y limpiando los campos de auditoría (`fecha_desactivacion=null`, `desactivado_por=null`).

Las rutas PATCH `/api/clientes/:id/desactivar` (con middlewares `verificarToken` y `esAgenteOAdmin`) y PATCH `/api/clientes/:id/reactivar` (con middlewares `verificarToken` y `esAdmin`) fueron configuradas en `cliente.routes.js`, registradas en `index.js` bajo el prefijo `/api/clientes`. El sistema implementa permisos diferenciados para desactivar y reactivar, asegurando un control centralizado sobre la reactivación de clientes.
