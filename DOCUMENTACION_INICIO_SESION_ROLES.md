# Historia de Usuario 1: Inicio de Sesión con Roles

La información detallada en la historia se puede hallar en el anexo correspondiente. Para el desarrollo de la historia 1, se creó la página `Login.jsx` utilizando React y Tailwind CSS para el diseño de la interfaz. La pantalla de inicio de sesión permite a los usuarios autenticarse mediante correo electrónico y contraseña, incluyendo validación en tiempo real y un botón para mostrar/ocultar la contraseña.

La interfaz fue realizada utilizando componentes de React con hooks `useState` y `useEffect`, la biblioteca de iconos Heroicons, Axios para peticiones HTTP y `jwt-decode` para decodificar tokens JWT. El componente `Login.jsx` implementa validación de formularios en tiempo real mediante expresiones regulares para el formato del email, y manejo de errores con mensajes descriptivos debajo de cada campo.

El componente incluye un `useEffect` que verifica al montar si existe un token válido en `localStorage`. Si existe, decodifica el token usando `jwt-decode` para obtener el rol del usuario y redirige automáticamente: administradores a `/admin`, agentes a `/agente`, y clientes a `/`. Al enviar el formulario, la función `handleSubmit` valida los campos, realiza una petición POST a `/api/auth/login` mediante Axios, y si es exitosa almacena el token JWT y datos del usuario en `localStorage`, dispara el evento `authChange` para actualizar componentes como el navbar, y redirige según el rol.

Para proteger las rutas del frontend, se creó el componente `RutaPrivada.jsx` que verifica la validez del token JWT y el rol del usuario mediante la función `verificarToken` de `tokenUtils.js`. Si el usuario no está autenticado o no tiene el rol requerido, redirige al login o a la ruta por defecto según el rol.

En el backend se utilizó Node.js con Express.js y Prisma como ORM para PostgreSQL. El servidor utiliza autenticación con JSON Web Tokens (JWT) para sesiones sin estado. Se creó el controlador `auth.controller.js` con la función `login` que procesa las credenciales del usuario.

El modelo `Usuario` fue definido en `schema.prisma` con el campo `rol` de tipo enum (`cliente`, `agente`, `admin`) y el campo `activo` (Boolean, default `true`) para controlar el acceso. La función `login` busca el usuario por email usando `prisma.usuario.findUnique`, verifica su existencia (retorna 404 si no existe), compara la contraseña con `bcrypt.compare` (retorna 401 si no coincide), verifica que el usuario esté activo (retorna 403 si está inactivo), y si todas las validaciones son exitosas genera un token JWT con `jsonwebtoken` incluyendo `id` y `rol` del usuario, expiración de 1 día, y firma con clave secreta de variables de entorno. El token y datos del usuario (sin contraseña) se retornan en la respuesta JSON.

Para proteger las rutas del backend, se crearon middlewares de autenticación y autorización. El middleware `verificarToken.js` extrae el token del header `Authorization` (formato `Bearer <token>`), verifica su validez con `jwt.verify`, busca el usuario en la base de datos para verificar existencia y estado activo, y agrega la información a `req.usuario`. Se crearon middlewares adicionales: `esAdmin.js` verifica rol de administrador, y `esAgenteOAdmin.js` verifica rol de agente o administrador. Estos middlewares retornan error 403 si el usuario no tiene el rol requerido.

La ruta POST `/api/auth/login` fue configurada en `auth.routes.js` sin middlewares de autenticación por ser un endpoint público, y registrada en `index.js` bajo el prefijo `/api/auth`. El sistema implementa redirección automática según el rol después de login exitoso: administradores a `/admin`, agentes a `/agente`, y clientes a `/`.
